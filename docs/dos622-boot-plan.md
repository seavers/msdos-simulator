# dos6.22.img 第一阶段启动实现步骤

## 1. 第一目标

本阶段只聚焦一个明确目标：

- 在网页中通过 `v86` 启动 `dos6.22.img`。
- 支持两种镜像来源：
  - 前端本地上传。
  - 服务端固定目录加载。

第一阶段不以“仙剑 95 可运行”作为交付边界，而是以“DOS 6.22 稳定启动到命令行”作为完成标准。

## 2. 固定目录约定

### 2.1 DOS 镜像固定目录

服务端固定目录约定为：

```text
storage/images/
```

推荐把镜像命名为：

```text
storage/images/dos6.22.img
```

### 2.2 v86 运行时资源

`v86` 的 `js/wasm` 资源由 `node_modules/v86` 提供，BIOS 文件放在：

```text
storage/runtime/v86/bios/
  seabios.bin
  vgabios.bin
```

## 3. 启动链路拆解

### 步骤 1：后端提供镜像清单与运行时状态

需要提供以下能力：

- 列出 `storage/images/` 下可启动镜像。
- 返回镜像元信息：
  - 文件名
  - 大小
  - Boot Signature
  - 推断驱动类型（floppy / hardDisk / cdrom）
- 返回 `v86` 运行时是否就绪：
  - `libv86.mjs`
  - `v86.wasm`
  - `seabios.bin`
  - `vgabios.bin`

### 步骤 2：前端支持两种镜像来源

页面需要支持：

- 选择“服务端固定目录镜像”。
- 选择“本地上传镜像”。
- 默认优先选中固定目录中的 `dos6.22.img`。

### 步骤 3：V86Adapter 接入真实启动流程

`V86Adapter` 需要真正完成以下初始化：

1. 动态加载 `/vendor/v86/libv86.mjs`。
2. 配置 `wasm_path`、`bios`、`vga_bios`。
3. 根据镜像类型挂载到 `fda` / `hda` / `cdrom`。
4. 把显示输出挂到页面中的 `canvas` / `screen container`。
5. 启动模拟器并把键盘焦点切到模拟器画面。

### 步骤 4：启动验证

最低验证标准：

1. 页面可识别到 `dos6.22.img`。
2. 点击启动后进入 BIOS / DOS 引导。
3. 最终能看到 `MS-DOS Version 6.22` 或 DOS 命令提示符。

## 4. 当前阶段实现范围

本轮实现包含：

- `docs` 文档整理。
- 服务端固定镜像目录支持。
- 镜像列表 API。
- `v86` 运行时状态检测。
- 前端固定镜像选择。
- `V86Adapter` 真实启动链路接入。

本轮不包含：

- 仙剑 95 的声音、显卡、内存专项兼容调优。
- save state。
- 虚拟磁盘写回。
- 游戏专用启动脚本注入。

## 5. 使用方式

完成本轮代码后，推荐按下面方式使用：

1. 把 `dos6.22.img` 放到 [storage/images/.gitkeep](./storage/images/.gitkeep) 同级目录下，命名为 `dos6.22.img`。
2. 启动本地服务。
3. 打开页面，镜像来源切到“服务端固定目录”。
4. 选择 `dos6.22.img`。
5. 引擎类型切到 `v86`。
6. 点击启动。

## 6. 下一阶段

当 `dos6.22.img` 稳定启动后，下一阶段会继续推进：

1. 自动挂载游戏盘。
2. 配置 `CONFIG.SYS` / `AUTOEXEC.BAT`。
3. 为仙剑 95 增加兼容预设。
4. 验证 SB16、VGA、键盘与定时器行为。

## 7. CPU 占用说明

`MS-DOS 6.22` 停在命令提示符时，常见现象是模拟器持续高占用 CPU。这不是页面绘制本身导致的，而是 DOS 在前台等待输入时通常不会自动进入高效 idle。

当前项目已经增加两层缓解方案：

1. 页面失焦时自动节能暂停。
2. 页面内提供手动暂停 / 继续控制，避免不需要运行时持续占满浏览器 CPU。

长期来看，如果希望 DOS 在“前台停在提示符但继续运行”的情况下仍保持更低 CPU，占用更稳妥的方案是在客体系统中安装 `DOSIdle` 一类 idle 工具，再结合模拟器侧的节能暂停策略。
